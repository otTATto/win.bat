# プロジェクト全体のルール

## コミュニケーション

- 応答や PR review、コメントアウトなどは日本語で行う
- 複雑なタスクでは実装前に計画を提示し、承認後に着手する

## コードスタイル

- ESLint/Prettier で処理済みの些末な指摘はしない
- エラーは握りつぶさず、適切にハンドリングする
- 重複実装をしない
  - まず既存の共通実装がないかを確認する
  - 類似実装が見つかった場合は、責務と依存関係を見て、適切な共有スコープに切り出すことを検討する
  - 無理にグローバル化しない
  - 共通化の影響範囲が大きい場合は、事前に方針を提示して承認を得る
- 後方互換性を保証しすぎない
  - 設計レベルで変更する場合は、その旨を提示し、承認後、もとの設計に関連するコードを残さない
  - 関連して、デッドコードは必ず削除し、決してコメントアウトで無効化しない
- 既存の動作するコードをリファクタリングする場合は、目的と影響範囲を明示する
  - その規模が大きい場合には、必ず事前の承認を求めること
- デバッグ用の `console.log()` は本番コードに残さない

## Git

- Conventional Commits に準拠したコミットメッセージを意識する
- 確認なしに自動で commit・push しない

## 禁止事項

- README・ドキュメントを勝手に生成・変更しない
- テストコードを確認なしに削除・変更・コメントアウトしない
- Linter や Formatter のルールを勝手に削除・変更しない
