# 変数名や関数名などの命名に関するルールや考え方

## 全般

- 読み手の連想を裏切らない命名を意識する
  - 例えば `getUserData()` 関数が get だけでなく set しているのは悪い命名である（あるいは、そもそも関数の分割が悪いかもしれない）
  - また DB I/O を伴うなど実行時間が長いことが予測されるのであれば `getUserData()` ではなく `fetchUserData()` などにすべきである
  - このように、似た意味の英単語であっても、その選定には妥当な理由が必要である
- 名前の長さはその影響範囲、つまりスコープの広さに依存する
  - `for` ループのインクリメント変数や `map` のコールバック関数に渡す変数など、スコープの狭い変数名は短くて構わない（例: `documents.map((doc) => {...})`）
  - 反対に、コンポーネントのトップレベルに宣言されるような変数や関数、複数ファイルから参照されうるものであれば、`data`, `info`, `item` のような抽象名は避ける
- 命名が長くなりすぎる場合にのみコメントアウトで補足する（コメントアウトを、命名の考慮をサボる言い訳に利用しない）
- 意味のある値は定数化する（マジックナンバーを避ける）

## 例

- フラグのニュアンスを持つ boolean 変数は、is, are, has, have, can, should などの状態動詞や助動詞から始める
- 定数は UPPER_SNAKE_CASE で記述する
  - ここでいう定数とは、再代入されない設定値・列挙値・共有される不変データを指す
  - Object や Map であっても、これらに該当する場合は UPPER_SNAKE_CASE とする
  - フレームワークで予約されている変数名（metadata 等）には適用しない
